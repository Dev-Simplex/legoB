# Snapping Engine

The highest-complexity subsystem in MVP. Story 2.3 is its introduction; Story 5.1 optimizes it.

## Contract

```typescript
export function computeSnap(args: {
  ray: THREE.Ray;
  hoveredPartId: string | null;
  activePartNumber: string;
  scene: Scene;
}):
  | { valid: true; position: [number, number, number]; rotationY: 0 | 90 | 180 | 270 }
  | { valid: false; reason: string };
```

## Algorithm (MVP baseline — Story 2.3)

1. **Ray vs. ground plane hit** → round `hit.x`, `hit.z` to nearest multiple of 10 LDU (half-stud). `y = 0`. Return valid.
2. **Ray vs. hovered brick's top surface:**
   - Look up stud positions for that brick's `partNumber` in `stud-positions.json`.
   - Transform local stud positions to world space using the brick's position + rotationY.
   - Find nearest world-space stud to the hit point.
   - If distance ≤ threshold (default 15 LDU ≈ 0.75 stud width) → snap to that stud.
   - `y = brick.y + brickHeight(partNumber)` (20 LDU brick, 24 LDU full-brick, 8 LDU plate — per part metadata).
3. **Collision check:** compute the active part's AABB at the proposed position. If it overlaps any existing part's AABB → return `{ valid: false, reason: 'collision' }`.
4. **No valid target** → `{ valid: false, reason: 'no snap target' }`.

## Stud positions lookup

**Source:** `src/data/stud-positions.json`, generated by `scripts/compute-stud-positions.ts` at build time.

**Generation logic:**
- For each part in `palette.json`:
  1. Fetch the `.dat` file from the pinned LDraw mirror (or bundled `public/ldraw-parts/`).
  2. Recursively resolve sub-file references (depth-first).
  3. Collect positions of every `stud.dat`, `stud3.dat`, `stud4.dat` reference (in local coords).
  4. Write `{ partNumber: [ [x,y,z], ... ] }` to JSON.

**Limitation (PO Risk AR3):** curved / angled / Technic parts may have studs the loader doesn't identify. MVP palette is deliberately restricted to flat/rectangular parts (bricks, plates, simple slopes). Document in README + Settings.

## Collision AABB

Use a pre-computed AABB per part (also in `stud-positions.json` for bundling). Compute once at build time by walking part geometry and taking the bounding box. In-scene collision = OBB-in-AABB intersection (rotate the active part's AABB by its `rotationY` then test overlap with each existing part's AABB).

For MVP simplicity, ignore rotation on collision (all bricks in 90° increments → AABB == OBB for flat/rectangular parts). Revisit when angled parts arrive post-MVP.

## Extensions post-MVP

- Connector-based snapping (LDCad-style connector DB) for Technic pins, hinges.
- Magnet-mode: snap preview previews the closest of multiple candidate studs.
- Undo/redo integration.
